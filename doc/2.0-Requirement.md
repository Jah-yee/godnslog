## 一、项目现状诊断

### 1.1 一句话结论

当前2.0开发严重偏离工程实际，存在**范围失控、实现质量极低、1.0功能丢失**三大核心问题。所有Phase标记"完成"但大量功能为**伪实现**（stub/placeholder），无法通过最小可用性验证。

### 1.2 核心问题清单

| # | 问题类型 | 具体表现 | 严重程度 |
|---|---------|---------|---------|
| 1 | 范围蔓延 | 从MVP直接跳跃到平台化（Phase 0~16全部标记完成），违背渐进式迭代原则 | 🔴 致命 |
| 2 | 实现质量低 | MCP Server: `apiCall`直接返回模拟数据；Workflow引擎: 仅有规则匹配，无任何动作执行；前端: 无详情页、无编辑、无WebSocket、无组件库 | 🔴 致命 |
| 3 | 1.0功能丢失 | 多用户角色系统仅存APIKey；标准DNS解析（A/CNAME/TXT/MX/NS）虽代码在但未在v2 API暴露；xip编码未在v2 API暴露；日志系统logrus仅旧代码使用 | 🔴 致命 |
| 4 | 架构割裂 | 1.0模型（TblDns/TblHttp）与2.0模型（Interaction）并存，无数据同步/迁移机制；API路由混乱（/api/record vs /api/v2/interactions） | 🟠 严重 |
| 5 | 前端Demo级 | 页面仅有列表展示，无详情/编辑/实时更新；shadcn/ui已引入但实际使用极浅；components目录为空；workflow/canary/evidence等页面只有占位 | 🟠 严重 |
| 6 | 测试不可信 | 单元测试存在但覆盖率不足；E2E测试只有登录和Dashboard；无API集成测试 | 🟡 中等 |
| 7 | 文档与实际脱节 | DEVELOPMENT_PLAN标记全部完成，但实际代码无法支撑任何声称的交付物 | 🟡 中等 |

## 二、重新梳理后的产品定位

### 2.1 一句话定位

**GODNSLOG 2.0 是面向安全团队、扫描器和 AI Agent 的自托管 OAST（Out-of-Band Application Security Testing）证据中枢**，负责生成可追踪 Payload、捕获多协议回连、自动归因、沉淀可解释证据，并通过标准化 API、CLI、MCP 和插件为外部工具赋能。

### 2.2 设计原则（保留并强化）

1. **自托管优先**：敏感交互数据不出组织，支持单机SQLite到生产MySQL/PostgreSQL的渐进部署。
2. **API先行**：前端、CLI、扫描器插件、MCP Server 共用同一套REST API与OpenAPI契约。
3. **证据优先**：每个Interaction必须可追溯到Case、Payload、Token、投放时间和测试人，支持时间线导出。
4. **协同优先**：不替代现有工具（Nuclei/Burp/Yakit/ZAP/xray/rad/CI/CD），而是增强其OAST能力。
5. **权限优先**：用户、APIKey、Agent 必须有最小权限、审计日志和高风险动作限制。
6. **Agent友好**：为AI Agent提供高层工具、异步等待、结构化证据、可恢复任务上下文和默认安全边界。

### 2.3 核心对象

| 对象 | 定义 | 当前状态 |
|------|------|---------|
| **Case** | 一次测试任务、漏洞验证或演练项目 | ✅ 数据模型+基础CRUD API有，但缺少批量操作、导入导出 |
| **Payload** | 带Token、场景、变量和生命周期的可追踪载荷 | ✅ 数据模型+基础CRUD API有，但模板渲染不完整，缺少预览 |
| **Interaction** | DNS、HTTP、SMTP、LDAP等外部回连事件 | ✅ 数据模型有，但**只有DNS/HTTP实际被监听**，其余为stub |
| **Evidence** | 由Payload、Interaction、时间线和备注组成的证据链 | ❌ 仅API占位，`v2GenerateEvidence`未实现生成逻辑 |
| **Workflow** | 命中后的通知、转发、打标和自动处理 | ⚠️ 规则引擎有匹配逻辑，但**无任何动作执行器** |
| **Canary** | 长期布设的诱饵Token，用于泄露/访问/横向移动监测 | ⚠️ 数据模型+API有，但**检测逻辑不完整** |

## 三、重新规划的需求清单

### 3.1 版本策略调整

原Roadmap从MVP直接跳跃到平台化，工程不可行。重新划分为**4个递进的发布里程碑**，每个里程碑必须达到"可上线、可验证"状态。

```
2.0-MVP  →  2.1-Scanner  →  2.2-Agent  →  2.3-Platform
(核心闭环)   (扫描器协同)    (MCP/Agent)   (平台化)
```

**关键调整**：
- 2.0-MVP 只保留真正闭环的功能，砍掉所有stub/placeholder。
- 每个里程碑必须有可运行的前端、可验证的API、可执行的CLI。
- 不追求"功能多"，追求"功能可用、体验完整"。

## 五、API与数据模型需求

### 5.1 API版本策略

| 版本 | 用途 | 状态 |
|------|------|------|
| `/api/v1/*` | 1.0兼容API，逐步废弃 | 保留但不新增功能 |
| `/api/v2/*` | 2.0主版本API | 当前开发重点，需补充完整 |

**必须补充的v2 API端点**（当前缺失或stub）：
- `PUT /api/v2/payloads/:id` - 更新Payload（当前只有Create/Revoke）
- `GET /api/v2/payloads/:id/interactions` - Payload关联的Interactions
- `GET /api/v2/cases/:id/interactions` - Case关联的Interactions
- `GET /api/v2/cases/:id/payloads` - Case关联的Payloads
- `GET /api/v2/interactions/stats` - 统计聚合（Dashboard用）
- `POST /api/v2/evidence/export` - 实际实现导出逻辑（当前只返回URL）
- `GET /api/v2/settings/*` - 系统设置API（域名、监听、通知）

### 5.2 数据模型统一策略

**方案选择（需用户决策，见问题清单Q3）**：

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A-双写同步 | 1.0 DNS/HTTP处理逻辑同时写入TblDns/TblHttp和Interaction | 兼容性好，查询统一 | 写入性能略降，需维护同步逻辑 |
| B-查询层统一 | 保留原表，v2查询时UNION两表数据 | 不改写入逻辑 | 查询复杂，分页/排序困难 |
| C-迁移废弃 | 一次性迁移历史数据到Interaction，废弃旧表 | 架构最清晰 | 风险高，需停机或双写过渡期 |

## 九、关键术语表

| 术语 | 英文 | 定义 |
|------|------|------|
| OAST | Out-of-Band Application Security Testing | 带外应用安全测试，利用外部服务器检测传统方法无法发现的漏洞 |
| OOB | Out-of-Band | 带外通信，指不通过主应用通道的外部网络交互 |
| SSRF | Server-Side Request Forgery | 服务器端请求伪造 |
| XXE | XML External Entity | XML外部实体注入 |
| SSTI | Server-Side Template Injection | 服务端模板注入 |
| DNS Rebinding | DNS重绑定 | 利用DNS TTL机制，使同一域名先后解析到不同IP的攻击/测试技术 |
| Canary Token | 金丝雀令牌 | 植入在系统中的诱饵，被访问时触发告警 |
| MCP | Model Context Protocol | Anthropic提出的AI Agent与外部工具交互的标准协议 |
| Payload | 载荷 | 用于触发漏洞的特定构造数据 |
| Interaction | 交互 | 目标系统与OAST服务器的外部网络交互事件 |
| Evidence | 证据 | 由Payload、Interaction、时间线组成的漏洞验证证据链 |
