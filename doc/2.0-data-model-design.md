# GODNSLOG 2.0 统一数据模型设计

> 文档版本: 2026-05-05-v1
> 基于决策项: Q1~Q7 已确认
> 设计目标: 消除1.0与2.0模型割裂，支持MVP核心闭环

---

## 一、设计原则

1. **单一事实源**：每个业务概念只对应一个数据库表，消除 `TblDns`/`TblHttp` 与 `Interaction` 并存。
2. **向后兼容**：1.0历史数据通过**一次性迁移脚本**导入新模型，迁移完成后1.0旧表标记废弃（只读）。
3. **前向可扩展**：字段设计预留2.1~2.3扩展空间（如 `Canary`、`RebindingRule`、`Workflow` 等）。
4. **JSON字段可控**：标签、变量、Headers等使用JSON列，但必须有明确定义的Schema，禁止无结构存储。
5. **审计追踪**：所有核心业务表（Case、Payload、Interaction、APIKey）含 `created_by`、`created_at`、`updated_at`。
6. **API模型共享**：所有后端接口的请求/响应结构体必须使用 Go `struct` 声明，放置于**独立可导出的包**（如 `internal/models/` 或 `pkg/apimodels/`）。这些结构体通过 `json` tag 控制序列化，客户端（Next.js/CLI）直接使用对应的 TypeScript 类型定义，确保服务端与客户端契约一致，避免接口漂移。

## 三、核心实体详细设计

### 3.1 User（用户）—— 保留1.0表结构

**表名**: `tbl_user`

|------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 用户ID |
| name | VARCHAR(64) | NOT NULL, UNIQUE | 用户名 |
| email | VARCHAR(64) | NOT NULL, UNIQUE | 邮箱 |
| role | TINYINT | NOT NULL, DEFAULT 0 | 0=super, 1=admin, 2=normal, 3=guest |
| short_id | VARCHAR(32) | NOT NULL, UNIQUE | 子域名标识，如 `userXXXX` |
| token | VARCHAR(128) | NOT NULL, UNIQUE | 1.0 API Token（向后兼容） |
| pass | VARCHAR(128) | NOT NULL | 密码bcrypt哈希 |
| lang | VARCHAR(16) | NOT NULL, DEFAULT 'en-US' | 语言偏好 |
| callback | TEXT | | 回调URL（1.0兼容） |
| callback_message | TEXT | | 回调消息模板 |
| rebind | JSON | | 用户级Rebinding配置 `["127.0.0.1", "127.0.0.2"]` |
| clean_interval | BIGINT | DEFAULT 3600 | 清理间隔（秒） |
| created_at | DATETIME | | 创建时间 |
| updated_at | DATETIME | | 更新时间 |

**索引**: `short_id`（DNS查询用）、`name`、`email`

**决策关联**: Q3确认保留1.0用户模型，前端废弃但数据保留。

### 3.3 Payload（可追踪载荷）

**表名**: `payloads`

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | VARCHAR(36) | PK | ULID/Base32编码 |
| case_id | VARCHAR(36) | NOT NULL, INDEX | 所属Case |
| token | VARCHAR(64) | NOT NULL, UNIQUE, INDEX | 追踪Token，如 `gdl_a1b2c3d4` |
| template | VARCHAR(64) | NOT NULL | 模板类型: SSRF/XXE/RCE/BLIND_SQLI |
| rendered_payload | TEXT | | 渲染后的完整Payload字符串 |
| variables | JSON | | 变量键值对 `{"target": "http://169.254.169.254"}` |
| status | VARCHAR(32) | NOT NULL, DEFAULT 'draft', INDEX | draft/deployed/hit/archived/expired |
| expected_protocol | VARCHAR(16) | | dns/http/smtp/ldap |
| expires_at | DATETIME | | 过期时间 |
| created_by | VARCHAR(36) | NOT NULL | 创建者UserID |
| created_at | DATETIME | | 创建时间 |
| updated_at | DATETIME | | 更新时间 |

**索引**: `token`（DNS/HTTP归因核心索引）、`case_id`、`status`、`expires_at`

**Token生成规则**: `gdl_` + 8位随机hex（原32位过长，缩短为8位hex = 4字节 = 32^8 ≈ 1万亿组合，足够安全且URL友好）。

**生命周期状态机**:
```
draft --(部署)--> deployed --(命中)--> hit --(归档)--> archived
  |                                              |
  +--(过期)--> expired <-------------------------+
```

### 3.4 Interaction（统一交互事件）

**表名**: `interactions`（取代 `tbl_dns` + `tbl_http`）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | VARCHAR(36) | PK | ULID/Base32编码 |
| type | VARCHAR(16) | NOT NULL, INDEX | dns/http/smtp/ldap/smb/ftp |
| case_id | VARCHAR(36) | INDEX | 关联Case（可为空，未归因时） |
| payload_id | VARCHAR(36) | INDEX | 关联Payload |
| token | VARCHAR(64) | INDEX | 触发此Interaction的Token |
| user_id | BIGINT | INDEX | 关联User（通过Token解析） |
| timestamp | DATETIME | NOT NULL | 事件发生时间 |
| source_ip | VARCHAR(64) | NOT NULL | 来源IP |
| source_port | INT | | 来源端口（HTTP/SMTP等需要） |
| domain | VARCHAR(255) | | 查询域名（DNS）或Host（HTTP） |
| dns_type | VARCHAR(16) | | A/AAAA/CNAME/TXT/MX/NS |
| method | VARCHAR(16) | | GET/POST/PUT...（HTTP） |
| path | TEXT | | 请求路径（HTTP） |
| headers | JSON | | HTTP请求头 `{ "User-Agent": "...", ... }` |
| body | MEDIUMTEXT | | HTTP请求体 |
| user_agent | TEXT | | User-Agent（HTTP） |
| content_type | VARCHAR(128) | | Content-Type（HTTP） |
| raw_data | MEDIUMTEXT | | 原始请求数据/报文 |
| meta | JSON | | 扩展元数据（如Rebinding标记、风险评分） |
| created_at | DATETIME | | 记录入库时间 |

**索引**: `type`, `token`, `case_id`, `payload_id`, `user_id`, `timestamp DESC`

**归因逻辑**:
1. DNS查询: 解析子域名提取 `token`，匹配 `payloads.token` → 填充 `payload_id` 和 `case_id`。
2. HTTP请求: 从域名/Path/Header中提取 `token`，匹配 `payloads.token` → 归因。
3. 未匹配: `case_id`/`payload_id` 为空，仅记录 `user_id`（通过子域名前缀匹配User.ShortID）。

**Meta字段Schema**（JSON）:
```json
{
  "is_rebinding": false,
  "rebinding_stage": 0,
  "risk_score": 0,
  "risk_signals": ["cloud_metadata_path", "internal_ip_access"],
  "classification": "suspicious_outbound"
}
```

> **决策关联**: Q3确认采用"双写同步"作为MVP过渡方案——1.0 DNS/HTTP Handler写入 `interactions` 表（同时保留写入旧表以保证1.0 API兼容，直到迁移完成）。

| 字段 | 类型 | 约束 | 说明 |

| id | VARCHAR(36) | PK | |

## 四、数据迁移策略

### 4.1 迁移方案（双写同步 + 一次性历史迁移）

```
Phase 1: 双写阶段（MVP开发期）
  - 1.0 DNS/HTTP Handler 继续写入 tbl_dns / tbl_http
  - 同时新增写入逻辑到 interactions 表
  - v2 API 查询 interactions 表
  - v1 API 继续查询 tbl_dns / tbl_http

Phase 2: 历史迁移（MVP发布前）
  - 执行一次性迁移脚本: tbl_dns → interactions, tbl_http → interactions
  - 迁移完成后标记 tbl_dns / tbl_http 为只读（或软删除）
  - 1.0前端废弃，v1 API标记废弃但保留

Phase 3: 清理（2.1版本）
  - 移除 tbl_dns / tbl_http 的写入逻辑
  - 可选：删除旧表（需备份）
```

### 4.2 迁移脚本要点

```sql
-- TblDns -> Interaction
INSERT INTO interactions (id, type, token, timestamp, source_ip, domain, dns_type, raw_data, created_at)
SELECT 
  generate_ulid(),
  'dns',
  var,
  ctime,
  ip,
  domain,
  'A',
  domain,
  atime
FROM tbl_dns;

-- TblHttp -> Interaction
INSERT INTO interactions (id, type, token, timestamp, source_ip, method, path, body, user_agent, content_type, raw_data, created_at)
SELECT 
  generate_ulid(),
  'http',
  var,
  ctime,
  ip,
  method,
  path,
  data,
  ua,
  ctype,
  path,
  atime
FROM tbl_http;
```

> **注意**: 迁移时需执行Token匹配，回填 `payload_id` 和 `case_id`。无法匹配的Interaction保持 `payload_id`/`case_id` 为空，仅保留 `user_id`。

## 六、Go模型代码规范

### 6.1 包结构

```
internal/
  models/          # 统一数据模型（所有v2表）
    user.go        # 包装TblUser，提供Role常量和方法
    case.go        # Case模型 + Request/Response
    payload.go     # Payload模型 + Variables类型 + Request/Response
    interaction.go # Interaction模型 + Headers类型 + 迁移方法
    resolve.go     # Resolve包装
    apikey.go      # APIKey模型（安全版）
    audit.go       # AuditLog模型
    settings.go    # Settings模型
    rebinding.go   # RebindingRule模型
    utils.go       # ID/Token生成工具
```

### 6.2 编码规范

- **表名**: 使用蛇形命名，2.0新表不加 `tbl_` 前缀（1.0保留表维持原名）。
- **字段名**: Go结构体使用驼峰，xorm tag使用蛇形。
- **ID生成**: 统一使用 `internal/models/utils.go` 的 `GenerateID()`（Base32编码的16字节随机数）。
- **时间字段**: 全部使用 `time.Time`，API响应层格式化为RFC3339字符串。
- **JSON字段**: 必须实现 `sql.Scanner` 和 `driver.Valuer` 接口（如 `Headers`, `Variables`）。
- **软删除**: 不使用物理删除，使用 `is_revoked` / `status=archived` 标记。

## 七、API模型对齐

### 7.1 当前问题

`models/v2.go` 中定义的API模型（`Case`, `Payload`, `Interaction` 等）与 `internal/models/` 中的数据库模型**字段类型不一致**（如 `v2.go` 中ID为 `string` 但内部为 `int64`）。

### 7.2 统一方案

**遵循"API模型共享"原则**：所有接口的请求/响应结构体必须在 Go 中以 `struct` 形式显式声明，放置在**独立可导出的包**中（推荐 `pkg/apimodels/` 或 `internal/models/`）。这些结构体既是服务端 Gin handler 的绑定目标，也是生成客户端 TypeScript 类型的唯一来源，确保前后端契约零偏差。

**具体执行**:
- 删除 `models/v2.go` 的冗余定义。
- 数据库实体（含 `xorm` tag）与 API 契约（含 `json` tag）共用同一 struct，通过 `json:"field,omitempty"` 控制 API 输出。
- 每个 struct 必须附带 `binding:"..."` tag（Gin验证），前端 Zod schema 与此严格对齐。
- 客户端（Next.js / CLI）通过代码生成工具（如 `quicktype` 或手写）从 Go struct 产出 TypeScript interface，禁止前端独立编造类型。

## 八、非功能性设计

### 8.1 性能

- `interactions` 表按 `timestamp DESC` 索引，支持时间范围查询。
- `payloads.token` 唯一索引确保归因查询 < 10ms。
- 大数据量时（>100万条Interaction），按月份分区或归档到冷表。

### 8.2 安全

- `api_keys.key_hash` 使用 bcrypt(cost=12) 存储。
- `interactions.body` 和 `raw_data` 在API返回时默认截断（>64KB截断），完整数据仅导出时提供。
- `audit_logs.details` 不存储敏感字段（如密码、Key明文）。

### 8.3 可维护性

- 所有表变更必须通过XORM Migration脚本（`migration.go`）管理。
- 新增字段必须有默认值，支持零停机迁移。
