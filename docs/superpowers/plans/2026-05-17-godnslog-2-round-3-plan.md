# GODNSLOG 2.0 Round 3 Plan

**定位**：本轮仍由 Codex 负责规划、设计、验收和实施准备，不负责代替执行开发。目标是把第二阶段的四份设计文档收敛为可实施、可验收、可排期的范围。

## 本轮目标

将当前的产品设计成果从“方向明确”推进到“可以开始实施”，重点解决三类问题：

1. **统一术语**：消除产品、API、Agent、控制面之间的概念漂移。
2. **统一优先级**：明确第一批必须落地的产品闭环，而不是并行铺开所有能力。
3. **统一范围**：形成下一轮实施的边界、依赖和验收口径。

## 本轮输入

- `docs/product-positioning.md`
- `docs/scanner-hub-adapter-design.md`
- `docs/agent-native-specification.md`
- `docs/unified-control-plane.md`
- 当前可通过的工程基线和 `/api/v2` 现状

## 本轮输出

本轮必须产出以下 4 项内容。

### 1. 统一术语与领域模型说明

形成一份统一说明，明确以下核心概念：

- Case
- Payload
- Probe
- Interaction
- Evidence
- Agent
- Agent Run
- Workspace
- Audit Event

这份文档的作用不是重复解释，而是规定“以后所有文档和实现都按这套定义说话”。

### 2. 首批 MVP 闭环范围

把当前大而全的产品能力，压缩成第一批必须落地的闭环：

1. 创建 Probe
2. 分发给 Scanner 或 Agent
3. 捕获 Interaction
4. 形成 Evidence
5. 导出结果并审计

本轮要明确这个闭环对应的最小页面、最小 API、最小 CLI/MCP、最小审计字段。

### 3. 官方支持边界与成熟度清单

输出一份官方支持边界，至少分成三档：

- **Primary**：首批官方主支持
- **Secondary**：官方提供脚本/桥接，但不承诺深度集成
- **Backlog**：已识别，但不进入当前周期

建议当前轮次按以下方向收敛：

- Primary: Nuclei, Burp Suite, Yakit/Yak
- Secondary: ZAP, xray/rad, Postman/Apifox
- Backlog: 其它扩展工具

### 4. 下一轮实施依赖图与验收标准

把后续实施前的依赖关系拉清楚：

- 哪些工作必须先做数据模型
- 哪些必须先做 API
- 哪些控制面可以并行
- 哪些 Scanner Hub 能力依赖 Agent 或 Evidence 先稳定

同时定义下一轮实施验收时必须回答的问题：

- 这个能力是否进入统一实体模型
- 是否进入统一审计链路
- 是否有明确角色和权限边界
- 是否进入首批 MVP 闭环

## 本轮工作主线

### 主线 A：术语与边界收敛

这是最高优先级。本轮如果不先统一术语，后续前后端、CLI、MCP、审计会出现多套定义并行。

### 主线 B：产品闭环压缩

目标不是继续设计更多页面，而是把 `OAST Evidence Hub` 的第一条闭环压缩到最小可交付。

### 主线 C：工具支持分级

目标不是“多写几个适配器”，而是先把官方支持边界写死，避免后续范围失控。

### 主线 D：实施前依赖梳理

目标是把后续真正要开发的内容变成几个清晰的包，而不是一个大而散的路线图。

## 本轮不做的事

为防止再次把规划阶段拖成实施阶段，本轮明确不做：

- 不直接改业务代码
- 不补写某个具体适配器实现
- 不直接开始前端页面开发
- 不扩展新的监听协议
- 不新增新的大特性线

## 预期决策

本轮结束后，必须能够明确以下决策：

1. GODNSLOG 2.0 的标准术语表
2. 第一批必须落地的产品闭环
3. 官方主支持工具名单
4. 首批控制面页面范围
5. 下一轮实施的阶段顺序

## 本轮验收标准

只有满足以下条件，本轮才算完成：

- 有一份统一术语文档，并能覆盖四份现有设计文档
- 有一份 MVP 闭环范围说明，明确首批交付边界
- 有一份工具支持分级清单，避免“支持列表无限增长”
- 有一份实施依赖图或实施顺序说明，可直接作为下一轮开发计划输入

## 建议的下一轮实施顺序

当本轮完成后，后续实施应优先按这个顺序推进：

1. 统一领域模型与 API 口径
2. 打通 Probe → Interaction → Evidence → Export 审计闭环
3. 落地首批控制面页面
4. 落地 Primary 工具集成
5. 落地 Agent 治理与回放能力
